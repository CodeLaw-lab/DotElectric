## 0. Аннотация

Документ содержит критический анализ архитектуры приложения DotElectric.TemplateEditor (WPF CAD-редактор, .NET 10, ~38K строк продакшн-кода + 1770 тестов). Этап 1 (44 FR) завершён, сборка 0/0, тесты зелёные. Архитектура в целом здоровая, но накопился ряд структурных «запахов», которые при реализации Этапа 2 (слои, экспорт, УГО, групповая работа) станут узким местом.

Документ преследует две цели:
1. **Зафиксировать** все найденные конструктивные замечания с доказательствами (`file:line`) и предлагаемыми решениями.
2. **Дать пошаговый план** рефакторинга, разбитый на атомарные рисковые спринты, готовый для превращения в тикеты.

Каждое замечание сопровождается оценкой **риска** и **приоритета** (P0 — критично, P1 — важно, P2 — желательно), чтобы упростить планирование.

## 2. Сильные стороны архитектуры (сохранять)

Перед тем как критиковать, фиксируем, что сделано хорошо — эти решения **нельзя ломать** при рефакторинге:

1. **Fixed-point в микронах** — все внутренние координаты `long`, сериализация без потери точности (`Models/Coordinate.cs`).
2. **Изоляция WPF-координат** — конвертация модель↔WPF только в `EditorCanvas` через `FromWpfPoint()/ToWpfPoint()`.
3. **MVVM + DI** — CommunityToolkit.Mvvm + Microsoft.Extensions.Hosting, `App.xaml.cs` корректно регистрирует сервисы.
4. **Undo/Redo Command-pattern** — `IUndoCommand` + `CommandHistory` с rollback при ошибке (`CommandHistory.cs:82-91`).
5. **Дисциплинированная документация** — Sprint-отчёты, AGENTS.md с Common Mistakes — высокий уровень зрелости команды.
6. **Тестовая инфраструктура** — 1760 тестов, абстракции `IDateTimeProvider`, `IDialogFileService` для детерминированных тестов.
7. **Изоляция WPF-диалогов** — `IDialogFileService`/`WpfDialogFileService` позволяет headless-тестирование.

## 3. Конструктивные замечания

### 3.1. Слой Models

#### 3.1.1. [P1] Избыточная трёхуровневая иерархия базовых классов

**Проблема:** Три уровня абстракции для одной цели:
```
ObjectBase (Models/ObjectBase.cs:7) — только Id + abstract Clone(): ObjectBase
   └── ModelBase (Models/ModelBase.cs:6) : ObjectBase, INotifyPropertyChanged
         └── TemplateObjectBase (Models/Objects/TemplateObjectBase.cs:11) : ModelBase
```

**Доказательства:**
- `ObjectBase` не имеет наследников кроме `ModelBase` — мёртвый уровень.
- Два разных контракта `Clone()`: `ObjectBase.Clone()` возвращает `ObjectBase` (никем не вызывается), `TemplateObjectBase.Clone()` возвращает `TemplateObjectBase` (ковариантный return, скрывает базовый).
- Комментарий в `ModelBase` («НЕ содержит WPF-зависимостей») фактический ложный: `INotifyPropertyChanged` из `System.ComponentModel` — это инфраструктура UI-биндингов.

**Последствия:** Запутанная иерархия, ложные комментарии, два неиспользуемых/конфликтующих контракта `Clone()`.

**Решение:** Схлопнуть в один уровень:
```csharp
public abstract class TemplateObjectBase : INotifyPropertyChanged
{
    public string Id { get; init; } = Guid.NewGuid().ToString();
    public abstract long MicronsX { get; set; }
    public abstract long MicronsY { get; set; }
    public abstract TemplateObjectBase Clone();
    // ...
    public event PropertyChangedEventHandler? PropertyChanged;
    protected void OnPropertyChanged([CallerMemberName] string? n = null) => PropertyChanged?.Invoke(this, new(n));
}
```
Удалить `ObjectBase.cs` и `ModelBase.cs`.

**Риск:** Средний — затрагивает все модели и сериализацию. Требует полного прогона round-trip тестов `.tdel`.

#### 3.1.2. [P1] Массивное дублирование INPC-бойлерплейта

**Проблема:** Каждый setter в `Line`/`Rectangle`/`Text` повторяет один и тот же паттерн:
```csharp
public long StartMicronsX
{
    get => _startMicronsX;
    set
    {
        if (_startMicronsX == value) return;
        _startMicronsX = value;
        OnPropertyChanged();
    }
}
```
В `Rectangle.cs` — 8 таких свойств (~50 строк бойлерплейта), плюс дублированные уведомления зависимых свойств (`RightMicronsX`, `CenterMicronsX` и т.д.).

**Доказательства:**
- `Line.cs:24-117` — 7 свойств по одному шаблону.
- `Rectangle.cs:23-136` — 8 свойств, из них 3 дополнительно вызывают `OnPropertyChanged(nameof(Right/Center...))`.
- `Text.cs` (446 строк) — аналогично.

**Последствия:** ~150 строк дублированного кода. Высокий риск забыть уведомить зависимое свойство (исторически такие баги были — см. Sprint 38/41/44 в AGENTS.md).

**Решение:** Использовать `[ObservableProperty]` source generator из CommunityToolkit.Mvvm:
```csharp
public partial class Rectangle : TemplateObjectBase
{
    [ObservableProperty]
    [NotifyPropertyChangedFor(nameof(RightMicronsX), nameof(CenterMicronsX))]
    private long _micronsX;
    // генерирует свойство MicronsX автоматически
}
```
CommunityToolkit.Mvvm не тянет WPF (только `System.ComponentModel`), допустимо на моделях.

**Риск:** Низкий — механическая замена, проверяется round-trip тестами.

#### 3.1.3. [P2] Дублирование и хрупкость `Clone()`

**Проблема:** `Line.Clone()`, `Rectangle.Clone()`, `Text.Clone()` вручную вызывают конструктор со всеми полями + инициализатор `Id = Guid.NewGuid()`. При добавлении нового свойства (как в Sprint 56 — цвета) легко забыть обновить Clone.

**Доказательства:**
- `Rectangle.cs:212-218` — перечисление 8 параметров.
- Исторически: при добавлении `StrokeColor`/`FillColor`/`Foreground` приходилось вручную править Clone.

**Решение:**
1. Покрыть Clone тестом «все public properties равны кроме Id» для каждого типа (защита от регресса).
2. Среднесрочно — ввести общий шаблон через `protected abstract T CreateInstance()` + copy-helper в базовом классе, либо через `[Cloneable]`-генератор.

**Риск:** Низкий.

### 3.2. EditorViewModel

#### 3.2.1. [P0] EditorViewModel — «фасад с пробросом» (1160 строк)

**Проблема:** EditorViewModel — индирекционный god object. Он не содержит логики (логика в 9 менеджерах), но содержит:
- ~60 forwarding-свойств вида `public double PanOffsetX { get => _zoomPanManager.PanOffsetX; set => ... }` (`EditorViewModel.cs:151-326`).

- `_previewHandler` (827-845), `_dirtyStateHandler` (848-863), `_toolManagerHandler` (866-871) — то же самое.

**Последствия:** Каждое новое свойство менеджера требует: (1) добавить свойство-проброс в EditorViewModel + (2) добавить case в switch-обработчик. Это хрупко и регулярно ломалось (см. Sprint 48 «Dirty indicator не появлялся» — забытый проброс).

<TextBox Text="{Binding ZoomPanManager.PanOffsetX, ...}" />

```csharp
public ZoomPanManager ZoomPanManager => _zoomPanManager;
// удалить ~60 свойств-пробросов и 4 switch-обработчика
```
Менеджеры уже `ObservableObject` — их PropertyChanged работает напрямую для WPF-биндинга.

**Оценка:** Уберёт ~60 свойств и ~80 строк switch-обработчиков. EditorViewModel сократится до ~700 строк чистой координации.

**Риск:** Средний — массовое изменение XAML-биндингов. Требует полного UI-копа и интеграционных тестов.

#### 3.2.2. [P1] Tools напрямую зависят от EditorViewModel

**Проблема:** `ToolManager(EditorViewModel)` создаёт инструменты через `new SelectTool(vm)` (`ToolManager.cs:16-24`). Все Tools (`SelectTool`, `ResizeTool`, `DrawingLineTool`, ...) принимают `EditorViewModel` напрямую — нарушение инверсии зависимостей. Tools могут дотянуться до чего угодно.

**Доказательства:**
- `ToolManager.cs:16-24` — `Func<EditorViewModel, ITool>`.
- `SelectTool.cs:19` — `private readonly EditorViewModel _editor;`

**Последствия:** Невозможно тестировать Tools в изоляции от всего EditorViewModel. Tools могут нарушать инкапсуляцию EditorViewModel.

**Решение:** Ввести интерфейс `IEditorContext` (или набор интерфейсов: `ISelectionContext`, `ICommandContext`, `IToolHostContext`). EditorViewModel реализует этот интерфейс:
```csharp
public interface IEditorContext
{
    TemplateObjectBase? SingleSelectedObject { get; }
    void PushCommand(IUndoCommand cmd);
    void SelectSingle(TemplateObjectBase obj);
    // ... только то, что нужно Tools
}
public class SelectTool : ITool
{
    public SelectTool(IEditorContext editor) { ... }
}
```

**Риск:** Средний — затрагивает все 6 Tools и их тесты.

#### 3.2.3. [P1] Неконсистентная группировка Undo

**Проблема:** Multi-операции группируются в `BatchCommand` непоследовательно:
- ✅ `DeleteSelected()` (`EditorViewModel.cs:1097-1102`) — группирует multi-delete.
- ✅ `PasteFromClipboard()` (`EditorViewModel.cs:620-626`) — группирует multi-paste.
- ❌ `MoveSelected()` (`EditorViewModel.cs:1110-1118`) — **НЕ группирует**: каждый объект → отдельная команда.
- ❌ `RotateSelectedClockwise()` (`EditorViewModel.cs:687-693`) — **НЕ группирует**.

**Доказательства:**
```csharp
// MoveSelected — нет группировки
public void MoveSelected(long deltaX, long deltaY)
{
    foreach (var obj in SelectedObjects.ToList())
    {
        var cmd = new MoveObjectCommand(obj, newX, newY);
        _commandHistory.Push(cmd);  // ← N отдельных команд
    }
}
```

**Последствия:** После nudge 5 объектов пользователь жмёт Undo 5 раз. Несоответствие UX между Delete/Paste и Move/Rotate.

**Решение:** Применить тот же `BatchCommand`-паттерн:
```csharp
public void MoveSelected(long deltaX, long deltaY)
{
    var objs = SelectedObjects.ToList();
    if (objs.Count == 0) return;
    if (objs.Count == 1) {
        _commandHistory.Push(new MoveObjectCommand(objs[0], ClampX(...), ClampY(...)));
        return;
    }
    var commands = objs.Select(o => (IUndoCommand)new MoveObjectCommand(o, ...)).ToList();
    _commandHistory.Push(new BatchCommand(commands, "Переместить объекты"));
}
```

**Риск:** Низкий — локальное изменение, проверяется unit-тестами CommandHistory.

#### 3.4.2. [P1] `CustomResizeCommand` — switch по типу вместо полиморфизма

**Проблема:** `CustomResizeCommand.Execute()`/`Undo()` — `switch (_object) { case Rectangle: ...; case Text: ...; case Line: ... }` (`CustomResizeCommand.cs:87-135`). Команда знает внутренности всех типов. Для Text поле `_newHeight` семантически переиспользуется как FontSize — misleading.

**Доказательства:**
```csharp
case Text text:
    text.MicronsX = _newX;
    text.MicronsY = _newY;
    text.FontSizeMicrons = _newHeight;  // ← _newHeight хранит FontSize!
    break;
```

**Последствия:** Нарушение OCP. Добавление нового типа объекта требует правки команды.

**Решение:** Полиморфная операция resize:
```csharp
// на объекте
public abstract void ApplyResize(ResizeState state);
public abstract ResizeState CaptureResizeState();
// в команде
public void Execute() {
    _object.ApplyResize(_newState);
    _markDirty?.Invoke();
}
```
Альтернатива (минимальная): переименовать поля для Text-случая или использовать отдельные параметры.

**Риск:** Средний — затрагивает ResizeTool и тесты.

### 3.6. Presentation (Tools, Behaviors, Views)

#### 3.6.1. [P1] `EditorCanvasBehavior` — 406 строк, статический кэш состояний

**Проблема:** Attached behavior смешивает: координатную конвертацию, маршрутизацию событий в Tools, контекстное меню, mouse capture, pan. Состояние каждого Canvas в `static ConditionalWeakTable<Canvas, EditorCanvasState>` (`EditorCanvasBehavior.cs:20`).

- Разбить на `CanvasInputRouter` (маршрутизация → ITool) + `CoordinateTransform` (чистые функции).
- Убрать статический кэш: хранить state в attached property `State`.

**Риск:** Средний — затрагивает горячий путь ввода.

#### 3.6.2. [P2] `ResizeTool` — 520 строк, сложная clamp-логика

**Проблема:** AGENTS.md (Sprint 49) описывает хитрый edge-based clamp. Логика resize-handle дублирует геометрию модели (`RightMicronsX`, `CenterMicronsX`).

**Решение:** Вынести математику клампа в чистые функции `ResizeMath`/`ResizeClamper`, тестируемые изолированно.

**Риск:** Средний.

---

#### 3.6.3. [P2] Code-behind `MainWindow.xaml.cs` — клавиатурные shortcuts захардкожены

**Проблема:** `Window_PreviewKeyDown` (`MainWindow.xaml.cs:18-60`) — switch по Key.V/L/R/T/E. Маппинг key→tool в code-behind.

**Последствия:** Невозможно переопределить shortcuts без перекомпиляции. Невозможно тестировать без UI-тестов.

**Решение:** Вынести маппинг в `ShortcutRegistry` (словарь key→tool), инжектируемый в Behavior. Code-behind только диспетчирует.

**Риск:** Низкий.

### 3.7. ValidationService

#### 3.7.1. [P1] Static god-service (537 строк), untestable/uninjectable

**Проблема:** `Helpers/ValidationService` — `static` (`ValidationService.cs:12`). Невозможно mock в тестах ViewModels/Services, которые его вызывают (`PropertiesViewModel`, `TemplateService`).

**Доказательства:** `PropertiesViewModel.cs:311` — `ValidationService.ValidateCoordinate` (прямой статический вызов).

**Решение:** Сделать `ITemplateValidator` (инстансный, инжектируемый) для domain-rules.

**Риск:** Средний — затрагивает все вызовы.

---

#### 3.7.2. [P2] Две ответственности в одном классе + дублирование HEX-логики

**Проблема:**
- Domain rules (`Validate(Template)`) и UI field validators (`ValidateCoordinate`, `ValidateHexColor`) в одном классе.
- `IsValidHexColor` (regex) и `ValidateHexColor` (return string?) — две реализации одной логики.

**Решение:**
- `ITemplateValidator` для domain-rules.
- `FieldValidators` (статические чистые функции) для UI — им инстансность не нужна.
- Удалить дубликат HEX.

**Риск:** Низкий.

### 3.8. Configuration & Build

#### 3.8.1. [P1] Нет Central Package Management

**Проблема:** Версии пакетов захардкожены в двух csproj (`CommunityToolkit.Mvvm 8.4.2`, `Microsoft.Extensions.* 10.0.9`, и т.д., `DotElectric.TemplateEditor.csproj:16-23`). Нет `Directory.Packages.props`.

**Последствия:** Дрейф версий между проектами, нельзя обновить одну зависимость централизованно.

**Решение:** Включить `<ManagePackageVersionsCentrally>true</ManagePackageVersionsCentrally>` + создать `src/Directory.Packages.props` со всеми `PackageVersion`.

**Риск:** Низкий.

#### 3.8.2. [P2] csproj дублируют Directory.Build.props

**Проблема:** `TargetFramework`, `Nullable`, `ImplicitUsings` объявлены и в `Directory.Build.props`, и в обоих csproj.

**Решение:** Убрать дубликаты из csproj, оставить только `OutputType`/`UseWPF`/`IsPackable`.

**Риск:** Низкий.

---

#### 3.8.3. [P1] `TreatWarningsAsErrors` только в CI

**Проблема:** Локальная сборка не ловит warnings — только CI `analyze` job (`/warnaserror`).

**Решение:** `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>` в `Directory.Build.props`.

**Риск:** Низкий — при текущем состоянии 0 warnings это безопасно.

### 3.9. Tests

#### 3.9.1. [P1] Перекос покрытия

**Проблема:** 165 тестов на конвертеры (stateless) vs 46 интеграционных тестов на весь lifecycle. 13 файлов `*Extended*`/`Additional*` — dumping-ground, выросли итеративно.

**Решение:** Дедуплицировать `*Extended*` (слить в родительские классы). Конвертировать бойлерплейт `[Fact]` в `[Theory]`+`[InlineData]`.

**Риск:** Низкий.

#### 3.9.2. [P1] Time-flaky assertions

**Проблема:** `TemplateTests.cs:158-159`, `TemplateExtendedTests.cs:80-81` — `Assert.True(created <= DateTime.Now)`. Гонка через полночь / медленный CI. `IDateTimeProvider` существует, но применяется непоследовательно.

**Решение:** Заменить `DateTime.Now` на инжектированный `IDateTimeProvider` во всех flaky-тестах.

**Риск:** Низкий.

#### 3.9.3. [P1] Нет coverage-gate в CI

**Проблема:** Coverage собирается и загружается как артефакт, но порога нет — падение до 10% пройдёт зелёным.

**Решение:** Добавить coverage-gate (Threshold ≥ 75%) + `reportgenerator` для HTML-отчёта + кэш NuGet.

**Риск:** Низкий.

## 8. Приоритеты реализации

**Делать обязательно (высокая ценность / низкий риск):**
- Весь **Спринт R1** — техдолг, который копится; каждый пункт — небольшой PR.
- **R3.1** (проброс через публичные менеджеры) — самый большой ROI: убирает ~60 свойств и 4 хрупких switch-обработчика, которые регулярно ломаются при добавлении свойств в менеджеры (Sprint 48).

**Делать с осторожностью (высокая ценность / средний риск):**
- **R2.1–R2.2** (реорганизация Models) — затрагивает сериализацию и все биндинги; нужен полный прогон round-trip `.tdel`.
- **R3.3** (DI-конструктор EditorViewModel) — затрагивает фабрику и все тесты EditorViewModel.

**Можно отложить (низкая срочность):**
- **R4.2–R4.3** (математика resize, shortcut-registry) — работают корректно, проблема только в расширяемости.
- **R3.5** (per-type PropertiesViewModel) — большой объём, окупится в Stage 2 при появлении кастомных свойств объектов.

**Не трогать:**
- Фиксированная точка в микронах, конвертация координат только в EditorCanvas — эти принципы корректны и хорошо задокументированы (AGENTS.md §«Architecture Must-Know»).
